# Architecture Notes

## Why FastAPI + n8n (not all-FastAPI or all-n8n)?

n8n owns **flow** — the message routing, retries, scheduled jobs, broadcast
loops. The shop owner can tweak these visually without redeploying code.

FastAPI owns **state and intelligence** — the database, Redis sessions, the
LLM agent with tools, the admin API. Keeping these in one Python service
means we have one place for migrations, one place for the tool registry,
and one place for tests.

n8n calls FastAPI via HTTP. FastAPI can also call n8n webhooks if it needs
to trigger a workflow (e.g. after a new conversation starts).

## Why server-side chat tokens?

WhatsApp messages are public to Meta — we never send tokens over them. The
chat token lives in Redis (24 h TTL) and is presented by n8n on every call
to `/api/ai/turn`. This means:

- Even if someone scrapes the customer's WhatsApp transcript, there's no
  internal credential to steal.
- The token gates LLM calls, so a leaked WhatsApp number alone can't drive
  up our LLM bill.
- Tokens rotate every 24 h naturally as conversations age out.

The token's hash is stored in `conversations.chat_token_hash`. The plain
token is only in Redis. If Redis is wiped, all active conversations are
forced to start fresh — that's the intended failure mode.

## Why three memory layers?

- **Redis (last 10 messages)** — fast, cheap, just enough to keep replies
  coherent within a single back-and-forth.
- **Postgres summary (last 7 days)** — a 200-token paragraph beats stuffing
  300 raw messages into the prompt. Generated by a nightly cron.
- **Qdrant embeddings (forever)** — for the rare "you helped me with the
  Samsung last March, do you have the same model?" — semantic search finds
  the relevant turn without dragging the whole history in.

Every customer turn pulls all three. Total prompt tokens stay bounded.

## Why tool-use, not free-text answers?

A free-text LLM will, eventually, quote a price that doesn't exist. The
fix isn't a longer system prompt — it's removing the option entirely.

The model has no way to surface a price unless `search_products` or
`get_product` returned it. The system prompt frames this as a hard rule;
the architecture enforces it by giving the model nothing else to work with.

## Why the intent classifier is a separate cheap model?

Two reasons:

1. **Cost.** Haiku at $0.80/MTok is ~10× cheaper than Sonnet. Most customer
   messages are greetings, goodbyes, or simple lookups — they don't need
   a 600-token reasoning model to process.
2. **Hard routing.** Some intents (`complaint`, `human_request`) should
   never go to the LLM at all. A pre-filter is the right place to make
   that decision deterministically.

Sonnet only runs on `product_inquiry` / `price_check` / `stock_check` —
the cases where its tool-use shines.

## Why category context on conversations?

If a customer asks "show me used Samsungs" and then says "what about under
50k?", the second message has no Samsung or "used" anywhere — but the
context should carry. Storing `category_context` on the conversation row
lets us seed the next `search_products` call with the sticky filter.

## What lives where (one-line each)

- `backend/app/main.py` — FastAPI app, mounts routers
- `backend/app/api/webhook.py` — Meta verification + signature check + inbound
- `backend/app/api/ai.py` — `/api/ai/turn` — the only LLM entrypoint
- `backend/app/services/agent.py` — tool-using Sonnet loop
- `backend/app/services/intent.py` — Haiku pre-filter
- `backend/app/tools/registry.py` — tool schemas + executors
- `backend/app/services/conversation.py` — token + memory + state transitions
- `backend/app/services/whatsapp.py` — Cloud API client + HMAC verify
- `admin-dashboard/src/app` — Next.js routes (one file per page)
- `n8n/workflows/*.json` — message flow, retries, schedules
